2022 *CTF-examination

发表于2022-04-20|更新于2022-04-20|pwn
|阅读量:

最近不是*CTF举办了吗，作为菜鸡的我一道题都没有做出来，最后还是靠着大佬的wp才明白是怎么一回事，记录一下

保护机制

checksec

保护全开了，有tcache，还是进ida看看

函数静态分析

main

main函数还是挺长的，不那种可以一眼看出漏洞的题，但不要着急，待我一点一点的分析

sub_1309

  • 程序初始化以及标题的打印

sub_1451

  • j老师菜单函数

sub_14EB

  • 添加学生，dword_503C存放着chunk的数量，从0开始增长，可以申请7次chunk
  • 使用的是calloc，每次申请会清空chunk(初始化)，一次申请两个chunk，一个 0x20，一个0x18，为了方便后面的讲解，我们将0x20的chunk写为 first_chunk，0x18的chunk写为second_chunk，其中first_chunk存放着second_chunk的地址，而first_chunk存放在qword_5080
  • 接着输入一个整数存放在second_chunk,这里先记为num1，后面会讲到

sub_1603

  • 给学生打分，buf[0]和0x80的取反再相与，这里就看到之前输入的数字num1有什么用了，是为了和buf[0]进行模运算，buf[0]%(num*10)，最后会把v2赋值给second_chunk + 4的位置
  • 这里发现v2是nsigned int，存在整型溢出，只要满足if ( *(_DWORD *)(qword_5080[i] + 24LL) == 1 )且v2 < 10就可以触发，v2 < 10好满足，只需要num1 = 1就可以了，if ( *(_DWORD *)(qword_5080[i] + 24LL) == 1 )的实现我们后面会讲到，先记着吧

sub_1760

  • 写下你对学生的评语，可以新建一个chunk存放评语，我们就叫做third_chunk吧，也可以在原来的基础上进行修改
  • third_chunk存放在second_chunk + 8的位置，对应的size存放在second_chunk + 16的位置，这里在申请的时候size不能超过0x400，再结合一共只能申请7次chunk，看来是想方设法限制我们对tcache的利用，真狗

sub_1948

  • 叫家长，接学生回家，释放chunk，挺干净的，没有利用点

sub_1F51

  • 切换角色

sub_1498

  • 学生菜单函数，默认从学生0开始操作

sub_1B0B

  • …毫无用处

sub_1D42

  • 判断*(_DWORD *)(qword_5080[a1] + 28是否为1,后面又发现了*(_DWORD *)(qword_5080[a1] + 28LL) = 1的赋值操作，看来每个chunk只能调用一次
  • 紧接着*(_DWORD *)(*(_QWORD *)qword_5080[a1] + 4LL) > 0x59u，熟不熟悉，就是前面sub_1603提到的整型溢出漏洞，可以利用上了吧，借助整型溢出，满足条件，可以打印出堆地址，从而得到任意堆地址，以及使任意chunk的size + 1，形成off-by-one
  • 最后判断(_QWORD *)(*(_QWORD *)qword_5080[a1] + 8LL)，也就是third_chunk是否存在，存在就打印其内容,可以想到泄露libc

sub_1EEE

  • 算数操作，将*(_DWORD *)(qword_5080[a1] + 24LL)与1异或，这就是最开始留下的问题，如何将*(_DWORD *)(qword_5080[i] + 24LL) == 1，如果忘记建议回去看看sub_1603

sub_1BE8

  • 这个函数感觉没啥用，就没有分析

sub_1FAE

  • 切换学生id，也就是更改对学生的操作，因为这个程序只能对单个学生进行操作，需要自己选择

GDB动态调试 + 解题思路

这里我们先上exp，后面跟着exp调试

首先先添加几个学生，因为first_chunk、second_chunk、third_chunk之间没有统一的bss段进行管理，所以它们都是按顺序排列的

接着看看qword_5080的内存情况，但是开了pie需要找到程序基地址，如下图红框所示

可以清晰地看见7个first_chnk的地址，其中第7个的地址和其他的first_chunk地址相差有点远是不是，这是为了后面修改地址做铺垫，后面会讲到

接下来执行pary()，也就是异或操作，满足整型溢出条件，从而执行check()，泄露堆地址

我们发现exp修改我们想要的地址时，后面居然*10，是不是很奇怪，别急，我们看看这个输入函数sub_13D5

发现没有，这里在输入的时候它会把我们输入的最后程序会判断，并在我们敲回车的前一个内容清0，因为数组是char，所以就相当于删除，这就会导致我们输入的chunk地址和我们得到的不符合，下面我们具体看看，这里我们输入的地址数据为94316454818608

这里我们我们可以清晰地看出，我们输入的94316454818608居然变成了9431645481860，少了一位，该怎么办呢，就直接将我们输入的数据*10就行了，现在输入948862438612960，我们再看看

这下我们现数据对了

下面我们成功将third_chunk1变成了off-by-one，就可以修改其紧邻third_chunk2的size。因为堆的地址都是紧紧相接的，所以我们只需要将third_chunk2的size地址低8位修改就可以了，为了让third_chunk2释放后可以进入unsort_bin，之前我们已经将third_chunk2的szie变成了0x400,我们需要让其大于0x418,这里我们填入0x91，使其变为0x491，如下图

成功送入unsort_bin，那怎样才能泄露出来呢，这时我们又想到了check()，也就是sub_1D42，是不是它除了可以打印堆地址,还可以打印comment，我们现在就需要构造一个fake_chunk，使其的comment指向main_arena + 96

在我们申请不在tcache_bin和fast_bin的堆块时，如果unsort_bin里有满足条件的chunk，程序会先在unsort_bin里进行切割，所以我们一点一点的把third_chunk2申请回来，并把main_arena + 96所在的chunk变为chunk5的comment

这里我们就可以看见我们伪造的chunk5了，现在就可以泄露main_arena + 96，从而得到__malloc_hook

泄露libc后，我们可以得到__free_hook以及system，我们的目的就是修改__free_hook为__system

先按照先前的方式，将chunk5的comment指针修改为__free_hook

然后修改__free_hook

最后执行free(4)就是执行system("/bin/sh")，getshell
